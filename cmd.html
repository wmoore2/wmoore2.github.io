William Moore
Introduction

CognisantMD (CMD) is a Toronto based startup in the medical technology space. They are the nation’s leading provider of EMR-integrated patient engagement solutions. Their main product is the Ocean platform. In my time working at CMD, I worked in a DevOps team, performing day-to-day tasks as if I was a full-time employee. I learned a great deal about the tech stack used at CMD, as well as DevOps culture.

About CMD

CognisantMD’s main product is the Ocean platform. Ocean is actually an acronym, standing for: Open Clinical Encounter Assistance Network, a place where clinicians could develop and share tools to improve patient care.

Ocean is setup as a SAAS business model and is hosted by CMD using a cloud provider. Ocean allows patients and healthcare professionals to securely share private health information for referrals, research, clinical use, and administration. Ocean provides many ways to increase patient engagement with your clinic. There is a tablet application as well as a website. Users will see the patient side of the application, which allows them to book appointments, sign up for reminders, and more. The clinician side of the application allows clinic staff to send patient reminders, create appointments, send e-referrals and more.

Ocean is integrated with many popular EMR (Electronic Medical Record) providers, meaning that information created with Ocean is shared with the EMR in use at the clinic.

Goals

My goals for this work term were largely focused around learning new technologies, and learning how to best work in a team.

I set a goal for myself to learn how to use the software tools used at CMD for DevOps. These tools include Terraform, Ansible, bash, and Nagios. I wrote plenty of Terraform, and really enjoyed it. Most of the work which I did involved Terraform in some way. I used Ansible to a lesser extent, mostly just out of necessity due to the nature of the tool. Bash and Nagios I used hand-in-hand with each other, and I was proficient at using these technologies. Overall, I met my goal of learning these new technologies.

Another goal I set for myself was learning to work better as a team. I had never worked on an agile team prior to this term, and so it was important for me to learn how to be a part of one. Over the course of the term I became more and more involved with the team and really learned through experience how to be a contributing member. I met this goal for sure.

My last goal was learning to manage my time more effectively. I accomplished this by making use of google calendar to time block my day. I would create a plan each morning for the tasks I wanted to accomplish. I was successful in reaching the goal of using my time better and noticed improvements right away starting this technique.

Job Description

My role at CMD consisted entirely of duties associated with DevOps. I was responsible for everything from writing Infrastructure as code to adding users to an allow list. Most of the more notable pieces of work I contributed to were in Terraform. These projects include DR migration, email warming, and overhauling user permissions.

The DR migration project was a huge undertaking. The basic idea was we wanted to migrate our disaster recovery site from MS Azure to AWS, as AWS is where we had all of our production code running. I contributed in writing a lot of the Terraform code for this migration, as well as in removing any remnants of Azure in our codebase after the migration was complete.

Overhauling user permissions was a great project to work on. The work involved with this included designing a way to programmatically assign users in AWS varying levels of permissions across AWS accounts within the same organization. I created a Terraform module which would assign users roles based on an array and permission set which was passed in. This allowed us to very accurately keep track of any changes made to the lists of who has what permissions. Given that Ocean needs to have great security, this was a very welcome change.

The email warming project was my favorite project to have worked on in my time at CMD. The preface of the project was this question: If we lose our production server, how would we be able to send email without our main IP? The answer was that we couldn’t if the IP was lost by AWS. The solution to this problem was simple, we needed another IP to fall back on. In practice, this is not so simple. Because of the ancient nature of email, it is filled with scammers and people sending garbage email. This means that there are strict filters applied to senders who send thousands of email, and Ocean send more than 70,000 emails on some days. We would surely trigger some red flags. We had an idea to write some code which would allow us to trickle email through a fresh ip, until it gained a positive reputation and was allowed through these filters. In this project I spent a lot of time writing the Terraform for the load balancer we were using for the email, as well as the terraform behind warming the email address.

Conclusions

This 8 month work-term was a wonderful experience. I greatly enjoyed my time working with the DevOps team at CMD, and making new friends there. The work was super enjoyable and I learned so much about DevOps – A field I knew very little about going into the work term. All in all, CognisantMD provided me with the tools, knowledge, and support I needed to succeed. Thank you CMD for a wonderful work term!
